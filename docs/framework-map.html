Apex Agent Orchestrator

How the framework fits together

AAO lets you run AI agents natively inside Salesforce: you describe an agent in configuration, grant it tools, and every conversation it has leaves a complete, queryable paper trail. This page has two views — the parts (every object and how they connect) and the process (what happens, step by step, when someone asks an agent a question).

1 · The parts — every object, and how they connect

Plain English: think of an agent as a new employee. The blueprint (left) is their job description, their toolbox, and the list of tools they're allowed to touch — all editable configuration, no code changes needed. The flight recorder (right) is the paper trail: every conversation, every question, and every single step the agent took gets its own record you can open and audit. The signals (bottom) are the office intercom — instant broadcasts that keep the engine moving and the screens live; they aren't stored.

2 · The process — what happens when you ask an agent something

Plain English: the agent works like a careful assistant on a loop: think → act → look at the result → think again, until it has enough to answer you. Every arrow between steps below is a hand-off through a platform event, which means each step runs in its own small transaction — a 40-step investigation is just as safe on the Salesforce platform as a 1-step one. And every loop has hard safety rails (right side), so an agent can never run forever or fail silently.

Where a run starts
Chat console (LWC) · Flow automation · Apex / API AgentChatController · AgentFlowRunAgentAction
⚡ Why the arrows matter: every hand-off is an Agent_Step_Event__e → fresh transaction. The queueable chain depth resets on each hop, so long runs never hit platform limits. The conversation travels between hops inside Agent_Run__c.History_Json__c.
1

You ask

Your message is saved, a new Agent Run record is created with status Running, and the first hop is queued. If it's part of an ongoing chat, the Session seeds prior turns into the conversation.

AgentEngine.runAgent() / runAgentInSession() → ExecutionLogger.startRun → AgentRunQueueable
2

The model thinks

The engine assembles the prompt — the agent's goal and rules, the catalog of tools it's allowed to use, and any relevant memories — compacts the history if it's grown long, then calls the LLM through a Named Credential. The request and reply are logged as a Step.

runLLMStep → ToolRegistry manifest + MemoryService.buildRecallBlock + HistoryCompactor → LLMClientFactory → OpenAI / Anthropic / Azure client
3

It answers in exactly one of three ways

The model must reply with a single JSON shape: use a tool, use several tools at once, or here's my final answer. Anything else is treated as a recoverable error and fed back.

{"tool":…} · {"tools":[…]} · {"final":…}
A

Run a tool

The Apex tool executes — query records, create or update data, describe an object, validate a field. The result is appended to the conversation and the loop returns to step 2. Parallel calls fan out one event per tool; the last to finish merges the results.

runToolStep → Type.forName(Tool_Class__c) → AgentToolResult · PendingToolCount__c countdown (FOR UPDATE)
B

Ask another agent

SubAgentTool starts a child run and the parent pauses — its step is marked Pending and nothing is scheduled. When the child finishes, its final answer wakes the parent, which resumes thinking at step 2.

startSubAgentRun (Depth__c ≤ 3) → child finishRun → resumeParentRun merges output
C

Final answer

The model is done gathering and commits to an answer. The loop ends and the run wraps up below.

{"final":{…}} parsed by runLLMStep
4

Wrap up

The run closes as Succeeded or Failed with the answer stored on the record. The session's turn count updates, worthwhile facts are captured as memories (async), and a UI event pushes the answer to the screen in real time.

ExecutionLogger.finishRun → SessionService.onRunFinished · MemoryService.enqueueCapture · UIEventPublisher → Agent_UI_Event__e

Safety rails

Every loop is bounded — an agent cannot run away or fail silently.

  • Step budget: the loop is capped per agent (default 10 steps). Agent_Definition__mdt.MaxSteps__c
  • Stuck-in-a-rut detector: the same tool called twice in a row with identical arguments forces a final answer. ExecutionLogger.hasIdenticalToolCall
  • Three strikes: 3 consecutive failed steps hard-fail the run; earlier failures are fed back to the model to recover from. ExecutionLogger.getConsecutiveErrorCount
  • Delegation depth: agents calling agents is capped at 3 levels. Agent_Run__c.Depth__c
  • Watchdog: a scheduled sweep fails runs that stall mid-flight. AgentWatchdogSchedulable (cron)
  • Full audit: every LLM call and tool call is written as an Agent Step — nothing is invisible. ExecutionLogger.logStep

Dashed green arrows are the loop: after a tool result or a sub-agent's answer comes back, the model thinks again with the new information — as many times as the step budget allows.

3 · Cast of classes — who does what

For the engineers in the room: the Apex classes behind the diagrams, one line each.

AgentEngine
The conductor. Only public entry point (runAgent / runAgentInSession); owns the LLM step, the tool step, and parent-run resume.
AgentRunQueueable · AgentToolQueueable
One async hop each — the LLM step (with callouts) and the tool step (deliberately no callouts).
AgentStepEventTrigger
Hears each Agent_Step_Event__e and enqueues the next hop — the trick that resets chain depth.
ToolRegistry
Reads the tool metadata, builds the manifest the model sees, resolves classes via Type.forName.
ExecutionLogger
Writes Runs and Steps, finishes runs, and is the resume choke point for suspended parent runs.
LLMClientFactory → OpenAI / Anthropic / AzureOpenAI clients
Provider adapters behind the LLMClient interface; swapped for LLMClientMock in tests so tests never hit a real LLM.
MemoryService · SalesforceMemoryProvider · MemoryCaptureQueueable
Recall memories into the prompt before a run; capture new ones asynchronously after it.
SessionService
Multi-turn conversations: seeds prior history into each new turn, updates the session when a run ends.
HistoryCompactor
Summarizes long conversation histories before they blow past heap and prompt limits.
UIEventPublisher
Fires Agent_UI_Event__e on step completion and run finish; failures are swallowed so the UI can never break the engine.
AgentWatchdogSchedulable · MemoryJanitorSchedulable
Scheduled crons: sweep stuck runs; prune expired or inactive memories.
The tools
QuerySalesforceTool · CreateRecordTool · UpdateRecordTool · DescribeObjectTool · ValidateFieldTool · SubAgentTool — all implement AgentTool and return AgentToolResult.
